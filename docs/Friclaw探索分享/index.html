FriClaw 探索分享

FriClaw 为什么值得做

今年很多人开始注意到 OpenClaw，不只是因为它“很强”，而是它第一次把 Agent 从零散能力点组织成了一条更完整的工作链路。

它让程序员看到模型还能怎样进入工作流，也让更多非程序员开始意识到：通用大模型不只属于聊天界面，它也可以被组织成一个持续帮人做事的系统。

OpenClaw 四大亮点

💬

接入真实沟通场景

不是只能在本地命令行里工作，而是能接到外部 IM，进入团队已经存在的协作入口。

♾️

持续工作

不是聊两句就结束，而是会话可以延续、状态可以承接、任务可以继续推进。

🧠

更强上下文

不是只看眼前一句话，而是带着历史和环境继续执行，更像一个在现场干活的系统。

🧰

可扩展能力

通过 Skill 与 MCP 把能力沉淀下来，而不是把所有逻辑都塞进一段 prompt 里。

为什么不是直接把 OpenClaw 部署到工作电脑

OpenClaw 很强，但对我来说也有现实顾虑：它需要的权限比较大，整体代码量也很大，我会天然担心系统边界和可控性问题。

换句话说，我想要的不只是“一个能做很多事的 Agent”，而是一个更符合自己使用方式、更容易把握住边界的系统。

PLAIN ENGLISH

不是从零再造一个完整 Agent 内核。

而是把已经很强的 Claude Code 当成执行核心。

在它之上补 IM 接入、上下文管理和后续扩展机制。

这样做，更贴近真实工作方式，也更可控。

💡
核心判断

FriClaw 不是为了复刻 OpenClaw，而是把 OpenClaw 代表的方向、Claude Code 已有的能力，以及我们自己的业务场景重新组织成一套真正适合自己的系统。

所以 FriClaw 的定位，就是一个“co-worker”

从 OpenClaw 的启发出发，站在 Claude Code 的执行能力之上，把 IM 接入、持续工作、上下文管理与能力扩展重新组织成一个真正能进入日常工作的 co-worker。

03

网关层、调度层、会话层

前面三层不直接负责“变聪明”，而是负责把外部消息变成一个可稳定运行的内部系统。

网关层：先统一入口，再谈后面的智能处理

对 FriClaw 来说，第一步不是“模型怎么回答”，而是“外部消息怎么稳定进入系统”。只有入口先统一，后面的调度、会话、Agent 执行和记忆承接，才有可能建立在一个稳定基础上。

🔌
建立连接

与企业微信、飞书、微信等平台建立接入关系，把系统真正放到工作入口里。

🔄
统一消息结构

把平台原始消息转成标准格式，让核心系统消费统一消息，而不是消费平台差异。

🎛️
保留平台特性

统一抽象不等于抹平差异。外层可以保留线程、卡片、富文本等平台能力，但这些差异不继续向内层扩散。

边界
关键边界

平台差异留在网关层，系统核心只消费标准消息。这样新增入口时，不需要重写 Agent 核心逻辑；调整平台交互方式时，也不需要改动会话和记忆系统。

调度层：让消息处理变得有秩序

1
Dispatcher 接管入口

它不是简单消息转发器，而是把外部平台事件转换成系统内部任务流程。

2
定位处理目标

根据消息归属定位目标会话，把外部事件真正纳入 FriClaw 自己的运行秩序。

3
Lane Queue 建立泳道

同一会话内严格串行，不同会话间完全并行，避免上下文串线和回复乱序。

4
让机制先于智能

模型再聪明，也不该承担消息乱序带来的系统性问题，这类问题应先由架构解决。

真正的处理单位不是 message，而是 session

只有把连续消息提升成会话对象，后面的上下文延续、工作空间复用、Claude Code 执行上下文恢复和历史状态继承，才有稳定落点。

04

Agent 执行层与记忆上下文层

Claude Code 在这里不是聊天接口，而是执行引擎；而记忆系统也不是聊天记录堆叠，而是可持续的上下文供给系统。

Claude Code Agent 在 FriClaw 里更像会话级执行引擎

到了 Agent 层，FriClaw 才真正进入“智能处理”。但这里最重要的区别是：Claude Code 承担的角色不是一个单纯回答器，而是一个会话级执行引擎。

🔁

延续上下文

在同一会话里持续承接状态，而不是每轮都从零开始组装一次 prompt。

🛠️

调用工具

在需要时调用工具、Skill 和 MCP，而不是只输出一段文字答案。

🌊

流式执行

输出的是 thinking、text、tool_use、done 等执行事件，而不是一次性黑箱结果。

♻️

可恢复

上下文过大或 session 过期时，不是简单丢掉，而是通过工作续接摘要平滑转移到新的执行上下文。

原上下文不可复用时，FriClaw 的目标不是“不断电”而是“不断线”

结
准确目标

FriClaw 追求的不是“永远保留一个不退出的进程”，而是：原上下文可复用时保持连续性，不可复用时通过摘要续接把工作平滑转移到新的执行上下文里。

FriClaw 的记忆系统，不等于把聊天记录全量拼回去

在很多聊天产品里，上下文基本等于历史消息拼接。但对 FriClaw 这种长期工作的系统来说，这种做法不够，因为系统还需要记住一些更长期、更稳定、对未来仍然有价值的东西。

🪪
Identity（SOUL.md）

定义系统身份、角色、行为边界和长期稳定的工作基线，保证它不是每轮都临时即兴决定自己是谁。

📚
Knowledge

沉淀用户偏好、联系人、项目背景等长期有效事实，让系统不只记住“这次说了什么”，还记住“以后还会有用什么”。

🎞️
Episode

保存历史会话的阶段性摘要与经验沉淀，不让历史无限膨胀，同时保留任务结果和关键结论。

PLAIN ENGLISH

Identity 定义长期稳定的助手行为基线。

Knowledge 沉淀用户和任务相关的长期事实。

Episode 压缩和回收历史会话经验。

真正值钱的不是“记得更多”，而是“记得更有结构”。

把主链路压缩成一句话

1网关负责适配平台，让消息稳定进入系统。
2调度层负责建立秩序，让会话处理从概率正确变成机制正确。
3会话层负责提供连续性，让系统处理单位从 message 提升为 session。
4Agent 层负责理解与执行，让 Claude Code 成为持续运行的智能执行单元。
5记忆层负责补充长期上下文，Skill / MCP 负责扩展外部能力边界。

最终总结

🏗️

不是单点能力堆叠

FriClaw 不是靠某一个超强模块完成全部工作，而是靠多个边界清晰的层次把一个通用模型组织成稳定系统。

📌

不是只解决一句话怎么回答

它真正解决的是：一个模型如何在真实系统里长期、稳定、可扩展地工作。

🤝

最终形态是 co-worker

当接入、调度、会话、执行、记忆和扩展能力被组织起来之后，系统才更接近一个真正进入团队日常工作的 co-worker。

一句话
FriClaw 的架构总结

它真正解决的，不是如何让模型回答一句话，而是如何让一个模型在真实系统里长期、稳定、可扩展地工作。
