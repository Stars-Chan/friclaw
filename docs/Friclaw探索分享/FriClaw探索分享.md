# FriClaw 探索分享

## 背景

今年最火的东西之一，应该就是 OpenClaw 了。它的影响力甚至已经不只停留在程序员或者 AI 圈子里，连很多非程序员群体也开始关注，甚至“养起龙虾来了”。

我觉得 OpenClaw 之所以会受到这么广泛的关注，不只是因为它能力强，更重要的是，它第一次把 Agent 从零散的能力点，组织成了一条相对完整的工作链路，并且以开源的方式呈现出来，让大家第一次比较具体地看到了 Agent 进入实际工作流的可能性。

它最重要的意义，不只是让程序员看到“模型原来还能这样用”，也让更多非程序员开始意识到：**通用大模型并不只能停留在聊天界面里，它也可以被组织成一个真正进入工作流、持续帮人做事的 Agent 系统。**

在我看来，OpenClaw 最值得关注的有四点：

- **能够接到外部 IM**：不是只能在本地命令行里工作，而是能进入真实沟通场景
- **能够持续工作**：不是简单聊两句就结束，而是会话可以延续、状态可以承接
- **有更好的上下文能力**：不是只看眼前一句话，而是能带着历史和环境继续执行
- **支持 Skill、MCP 扩展**：能力不是写死的，而是可以不断增加、沉淀和组织

但对我来说，OpenClaw 虽然很强，我却一直没有直接在自己的工作电脑上部署它。原因也很现实：一方面它需要的权限比较大，另一方面整体代码量也很大，我会天然担心系统边界和可控性问题。说得直接一点，就是我很担心它把我的工作环境搞崩，而且我自己又很难完全把握住它。

也正因为这样，我当时就在想：能不能围绕这四个核心能力，做一个更适合我自己使用方式的 Claw？

后来我和昕哥讨论过一个很关键的问题：OpenClaw 能做的很多事情，其实 Claude Code 本身也已经能做了。既然如此，那是不是可以换一种思路——不是从头去造一个完整的 Agent，而是基于 Claude Code 这个已经很强的执行内核，往上补我们真正需要的能力，比如 IM 接入、上下文管理，以及后续按需增加的扩展机制。

于是，FriClaw 就是在这个思路下开始的。某种意义上说，它就是一个站在 Claude Code 这个“巨人”肩膀上的 Claw。

FriClaw 不是为了复刻 OpenClaw，而是希望把 OpenClaw 所代表的方向、Claude Code 已经具备的能力，以及我们自己的业务场景和工作方式，重新组织成一套真正适合自己的系统。

也正因为如此，FriClaw 从一开始就不是把自己定位成“另一个聊天机器人”，而是定位成一个真正能进入日常工作、替我们持续干活的 **co-worker**。

所以，FriClaw 当前架构的出发点，并不是“从零设计一个全新的 AI 框架”，而是沿着 OpenClaw 已经证明过的方向继续往前走，并建立在 Claude Code 这样强大的 Agent 能力之上：把外部接入、持续执行、上下文管理和能力扩展这几件事，做成一套真正适合我们自己的系统。

也正因为如此，FriClaw 当前的架构，本质上不是围绕“单次问答”组织的，而是围绕“一个 Agent 系统如何稳定运行，并真正成为团队 co-worker”来组织的。

## 一、整体架构总览：先把系统骨架搭起来

FriClaw 当前架构其实可以理解成一条很清晰的链路：**外部入口负责接入消息，调度层负责组织运行，会话层负责承接状态，Agent 层负责理解和执行，记忆与扩展层负责让系统持续工作和持续生长。**

如果把它画得更直观一点，大致就是这样：

```text
企业微信（同步支持飞书、微信）
        ↓
      网关层
        ↓
 调度层（Dispatcher / Lane Queue）
        ↓
   会话层（SessionManager）
        ↓
 Claude Code Agent
   ↙        ↓        ↘
Memory    Skill     MCP
        ↓
   响应回到企业微信
```

这里每一层都不是为了“分目录”而存在，而是在回答一个明确的问题：

- **网关层**：消息从哪里来、以什么格式进入系统
- **调度层**：消息应该如何排队、如何分发、按什么顺序执行
- **会话层**：消息归属到哪个会话、上下文如何承接、状态如何隔离
- **Agent 执行层**：谁来理解任务、调用能力、完成执行
- **Memory**：系统如何把历史、记忆和上下文组织起来，支撑 Agent 持续工作
- **Skill / MCP**：Agent 在执行过程中可以调用哪些原生能力和扩展能力

## 二、网关层：先统一入口，再谈后面的智能处理

对 FriClaw 来说，网关层之所以放在最前面，是因为一个真正要进入工作流的 Agent 系统，第一步不是“模型怎么回答”，而是“外部消息怎么稳定进入系统”。只有入口先统一了，后面的调度、会话、Agent 执行和记忆承接，才有可能建立在一个稳定的基础上。

### 1. 网关层不做业务判断，只做平台适配

网关层并不负责理解用户意图，也不负责管理复杂上下文。它更像系统最外层的“翻译器”和“适配器”，核心职责只有四件事：

- 建立与外部平台的连接
- 接收平台事件
- 把平台原始消息转换成统一结构
- 把系统内部响应翻译回平台原生格式

这个边界看起来很朴素，但其实非常关键。因为一旦让平台层承担过多业务判断，很快就会出现两个问题：

- 平台差异开始污染核心逻辑
- 后续新增入口时，业务逻辑被重复实现

所以 FriClaw 在这里的边界很明确：**平台差异留在网关层，系统核心只消费标准消息。**

这样拆开的价值也很直接：

- 新增入口时，不需要重写 Agent 核心逻辑
- 调整平台交互方式时，不需要改动会话和记忆系统
- 核心执行层只处理统一消息，不处理平台特有细节

### 2. 统一入口，不等于抹平差异

网关层的目标是统一入口，但不是把所有平台都硬做成一样。更准确地说，它做的是**统一抽象、保留差异**。

比如：

- 飞书可以用卡片更新实现更自然的流式输出
- 企业微信没有同样的流式卡片能力，就采用分块消息来模拟流式效果
- 飞书话题群里的 thread 可以映射成更细粒度的会话
- 不同平台对交互式问题、图片、富文本的支持程度也不同

也就是说，网关层做的不是消灭差异，而是吸收差异：允许外层保留平台特性，但不让这些差异继续往内层扩散。

### 3. 网关层的意义，不只是接消息

从更高一层看，网关层其实同时决定了两件事：

- 对外，FriClaw 能出现在哪些工作入口里
- 对内，系统能不能建立一条统一、稳定的处理链路

所以这一层虽然不直接承担“智能”，但它决定了整个系统是不是可接入、可扩展、可持续演进。

## 三、调度层：让消息处理变得有秩序

如果说网关层解决的是“消息怎么进来”，那么调度层解决的就是另一个关键问题：**消息进来之后，应该按什么顺序、用什么机制被处理。**

在 FriClaw 当前设计里，调度层主要由两个部分组成：

- **Dispatcher**：负责把外部消息纳入系统内部处理流程
- **Lane Queue**：负责保证同一会话内的消息按秩序执行

### 1. Dispatcher：系统里的调度中枢，而不是消息转发器

从表面上看，Dispatcher 像是在做“转发”：收到消息，找到目标，再交给后面的执行链路。但如果只把它理解成转发器，会低估它在架构里的位置。

Dispatcher 真正做的，是把“外部平台事件”转换成“系统内部任务流程”。也就是说，在网关层拿到的还是平台消息；到了 Dispatcher 这一层，消息才真正被纳入 FriClaw 自己的运行秩序里。

它负责的事情包括：

- 根据消息归属定位处理目标
- 把消息投递到对应会话的处理通道
- 对部分系统命令做前置短路处理
- 保证外部平台不会直接触碰内部实现

所以 Dispatcher 的意义，不是简单转发，而是把系统入口和内部运行机制真正接起来。

### 2. Lane Queue：保证系统稳定运行的，首先是调度秩序

调度层里另一个很关键的点是 Lane Queue。

它解决的问题不是“怎么调用模型”，而是“怎么避免同一会话里的消息互相打架”。这个问题在会话型系统里非常真实：

- 用户连续发送多条消息
- 平台重复投递或短时间内多次触发
- 一条消息还没处理完，下一条又到了

如果没有调度机制，很容易出现：

- 上下文串线
- 回复顺序错乱
- 后发消息覆盖前一轮状态
- 同一个会话并发触发多个 Agent 执行过程

Lane Queue 的设计，本质上是在系统层引入一个简单但有效的约束：

```text
同一会话内：严格串行
不同会话间：完全并行
```

这个约束决定了 FriClaw 的运行方式不是“谁先抢到资源谁执行”，而是“每个会话有自己的处理泳道”。

### 3. 调度层的价值：不是为了排队，而是为了建立确定性

很多时候我们会把队列理解成削峰填谷或者控制并发，但对 FriClaw 来说，调度层更大的价值其实是：**把会话处理过程从概率正确，变成机制正确。**

也就是说，模型再聪明，也不应该承担消息乱序带来的系统性问题；这种问题应该先由架构解决。只有调度秩序先建立起来，后面的智能执行才不会建立在不稳定输入之上。

---

## 四、会话层：把消息组织成可持续的工作上下文

如果说调度层解决的是“怎么处理”，那么会话层解决的就是“处理结果接到哪里”。

在 FriClaw 当前设计里，会话层主要由 **SessionManager** 承担。它的意义在于：系统处理的不是一条条彼此独立的 message，而是一段持续存在、带有状态、可恢复的 session。

### 1. SessionManager：把连续对话从消息列表提升成会话对象

一个FriClaw和普通消息回调程序最大的区别，就是它不能把每条消息都当成独立请求。

SessionManager 的存在，意味着系统处理的不是零散消息，而是一段持续存在、带有状态、可恢复的会话。这背后其实有一个很重要的架构判断：**对FriClaw来说，真正有意义的处理单位不是 message，而是 session。**

因为只有引入会话这个层次，后面这些事情才有落点：

- 上下文如何延续
- 哪个工作空间归属于哪个用户或聊天
- 哪个 Claude Code 执行上下文应当继续复用
- 哪些历史状态应当跟着本轮继续生效

### 2. 会话层真正承接的是状态连续性

如果没有会话层，整个系统很快就会退化成“每条消息调用一次模型”的松散结构，这显然支撑不了 FriClaw 这种目标。

也正因为有会话层，FriClaw 才能把一次次消息处理，组织成一个持续工作的过程：同一个用户、同一个群聊、同一段上下文，不再只是消息列表，而是一个可以被识别、复用、恢复和清理的运行单元。

从这个角度看，会话层真正解决的不是“存一下聊天记录”，而是给后面的 Agent 执行、记忆供给和工作空间隔离，提供一个稳定的承载点。

---

## 五、Agent 执行层：Claude Code 在这里不是聊天接口，而是执行引擎

到了 Agent 层，FriClaw 才真正进入“智能处理”的部分。

但这里有一个很重要的区别：在当前架构里，Claude Code 承担的角色，并不是一个单纯的回答器，而是**会话级执行引擎**。

这也是 FriClaw 和很多简单 AI 接入方案最大的不同之一。

### 1. Agent 层解决的，不只是回答问题

如果系统目标只是“一问一答”，那么直接请求模型、拿到文本、返回平台，其实就够了。

但 FriClaw 当前设计显然不是这样。它希望 Agent 具备的是更持续的工作能力：

- 在同一会话里延续上下文
- 在多轮交互里保持状态连续
- 在需要时调用工具
- 以流式方式输出内容
- 在一轮结束后保留可恢复的执行轨迹

所以 Claude Code Agent 在这里更像一个**长期驻留的会话执行单元**，而不是“调用一次就结束”的 API 包装层。

### 2. 一个会话对应一个执行上下文

当前设计中，一个 conversationId 会尽量复用同一个 Claude Code 执行上下文，并通过 `session_id` 维持恢复能力。这样做的重点不是“实现复杂”，而是为了让同一个会话在多轮交互中保持连续性。

换句话说，FriClaw 真正关心的，不是每一轮都单独拿到一个答案，而是让同一个会话始终像“同一个助手”在持续工作。

不过，这个执行上下文也不是永久存活的。

它的核心思路是：当原来的执行上下文不再适合继续直接复用时，不是简单丢掉会话，而是先生成一份“工作续接摘要”，再在新的 Claude Code 进程里基于这份摘要继续工作。

这套能力大致有两个典型入口：

- **场景 A：上下文过大，主动压缩**。当估算会话上下文接近 Claude Code 最大 token 的 0.7 时，先生成工作续接摘要，再结束旧的执行上下文，并在新的 Claude Code 进程里基于摘要继续。
- **场景 B：session 过期，被动恢复**。当 session 被回收、Claude Code 子进程因 idle timeout 释放，甚至 daemon 重启导致原执行上下文消失时，系统读取最近一次工作续接摘要，重新拉起新的 Claude Code 进程，并把摘要注入进去继续服务。

所以更准确地说，FriClaw 追求的不是“永远保留一个不退出的进程”，而是：**在原上下文可复用时保持连续性，在原上下文不可复用时通过摘要续接把工作平滑转移到新的执行上下文里。**

从这个角度看，Agent 层做的不是“临时生成响应”，而是“持续承载会话执行”。

这带来几个直接的架构收益：

- 同一会话的工作状态不需要反复从零拼装
- 多轮交互的状态切换更自然
- 工具调用与追问可以被组织成连续过程
- 系统可以更稳定地管理长会话

### 3. 流式交互是执行层的一部分，不只是展示优化

FriClaw 当前设计里，Agent 输出不是一次性文本，而是会产生 thinking、text、tool_use、done 等事件流。

这件事很值得强调，因为它说明系统并不是把 Agent 当成“黑箱返回最终字符串”，而是把它当成一个持续发出执行事件的引擎。

一旦用这种方式组织，前后的架构关系就会变得更清楚：

- 网关层负责把这些事件翻译成平台可展示的形式
- 调度层负责保证事件属于正确的会话
- Agent 层负责产生执行过程本身

也就是说，流式并不是 UI 层的小优化，而是整个系统执行模型的一部分。

### 4. Claude Code 在 FriClaw 里，更像一个“可被系统组织的智能核心”

FriClaw 当前架构没有让 Claude Code 单独暴露在系统中心，而是把它放进了一个更完整的运行框架里：

- 前面有会话和调度机制约束输入
- 后面有记忆、工作空间和 MCP 能力提供支撑
- 外面还有多平台网关负责承接与呈现

所以从架构上看，FriClaw 做的并不是“接入一个 Claude”，而是**把 Claude Code 变成系统中的一个可持续运行、可恢复、可扩展的智能执行单元。**

## 六、记忆与上下文层：让系统不是只看这句话，而是带着状态继续工作

如果说 Agent 层解决的是“谁来执行”，那么记忆与上下文层解决的就是“执行时凭什么知道自己现在处在什么状态”。

这一层是 FriClaw 当前架构里非常关键的一层，因为FriClaw和普通聊天系统最大的差异，往往就体现在这里。

### 1. FriClaw 不是把上下文简单理解成聊天记录

在很多简单的聊天产品里，“上下文”基本等于历史消息拼接。但在 FriClaw 当前设计中，这种理解是不够的。

因为一个真正可工作的FriClaw，不只是需要记住用户刚刚说过什么，还需要记住一些更长期、更稳定、对未来仍然有价值的东西。

从当前设计上看，FriClaw 把记忆拆成了三层：

- **Identity（SOUL.md）**：系统身份与长期行为基调
- **Knowledge**：用户知识、偏好、项目相关信息
- **Episode**：历史会话摘要与经验沉淀

这种分层方式背后的思路很清楚：

- 不是所有信息都应该以同样方式保存
- 不是所有历史都值得每次完整重放
- 比起“记得更多”，更重要的是“记得更有结构”

从实现上看，这套记忆系统也不是单一存储，而是由三部分共同组成：

- **SQLite**：作为记忆索引和全文检索后端
- **Markdown 文件**：作为记忆的原始存储形式
- **MCP tools**：作为 Claude Code 访问记忆的接口层

### 2. Identity 层：先确定这个系统是谁

Identity 层听起来很抽象，但它在FriClaw里其实非常重要。

因为 FriClaw 不是一个裸模型，而是一个带有明确身份设定的系统。它的风格、角色、行为边界，并不是每轮由模型即兴决定，而是在每次创建 Claude Code 进程时，以 SOUL.md 的形式作为系统级提示持续注入。

这一层的意义不在于“增加人设感”，而在于给整个系统建立长期稳定的行为基线。

换句话说，它解决的是：

- FriClaw应该以什么方式工作
- 它面对用户时的默认行为倾向是什么
- 哪些原则应该跨会话持续生效

### 3. Knowledge 层：沉淀的是用户相关事实，而不是一次性聊天噪声

Knowledge 层更偏向长期有效信息，比如用户偏好、联系人、项目背景、一些会被反复用到的知识。

这一层很关键，因为它把“会话记忆”往前推进了一步：不再只是记住这次说过什么，而是开始沉淀“以后可能继续有用的信息”。

从系统角度看，这意味着 FriClaw 不只是一个会聊天的 Agent，而是一个可以逐渐积累用户知识的系统。

这也是它能从一次次对话里慢慢形成稳定工作方式的基础。

### 4. Episode 层：保留历史，但不让历史无限膨胀

长期会话系统一定会遇到一个问题：历史会越来越长。

如果所有历史都原样堆给模型，会遇到几个非常现实的问题：

- 成本持续上升
- 噪声越来越多
- 当前真正重要的信息反而被淹没

FriClaw 当前设计没有走“完整重放所有历史”的路线，而是通过 Episode 这种摘要化的方式保存会话阶段性结果。

这个思路很合理。因为对长期运行的助手来说，最有价值的往往不是原始聊天记录本身，而是从聊天中提炼出来的任务结果、关键信息和阶段结论。

### 5. 记忆层真正解决的，是“可持续的上下文供给”

把三层记忆放在一起看，会更容易理解这层的真实作用。

```text
Identity：定义长期稳定的助手行为基线
Knowledge：沉淀用户和任务相关的长期事实
Episode：压缩和回收历史会话经验
```

这意味着 FriClaw 在给 Agent 提供上下文时，不是只依赖短期聊天历史，而是可以把长期身份、已沉淀知识和历史摘要一起组织起来。

这件事的架构价值很大，因为它把“上下文”从一段临时文本，提升成了一套可管理的系统能力。

### 6. 上下文层和会话层是相互配合的

值得注意的是，记忆层并不是孤立工作的。

- 会话层负责定义“这一轮属于哪个连续过程”
- 记忆层负责决定“这个连续过程在更长时间尺度上可以继承什么”

两者配合之后，FriClaw 才真正具备一种FriClaw该有的连续性：

- 当前轮次不脱离眼前问题
- 同一会话不丢状态
- 跨会话又能沉淀长期记忆

这比单纯把聊天历史交给模型，要稳定得多，也更像一个真正长期工作的系统。

---

## 七、总结：一条请求如何穿过整套架构

如果把前面几层重新串起来，FriClaw 当前架构的主链路其实可以概括成下面这样：

```text
1. 用户在 企业微信 发来消息
2. 对应网关接收消息，并转换成系统统一消息结构
3. Dispatcher 接管这条消息，定位或创建对应会话
4. 消息进入该会话的 Lane Queue，保证会话内串行处理
5. SessionManager 为这次处理提供会话环境与工作空间
6. Claude Code Agent 在该会话上下文中执行本轮任务
7. 执行过程中按需读取记忆、使用工具、调用 MCP 能力
8. Agent 产出流式事件或最终结果
9. 网关把结果翻译回平台原生形式并返回给用户
```

如果只看步骤，这条链路并不复杂；但它真正重要的地方在于，每一层只做自己该做的事情：

- 网关负责适配平台
- 调度层负责建立秩序
- 会话层负责提供连续性
- Agent 层负责理解与执行
- 记忆层负责补充长期上下文
- 扩展层负责提供外部能力

这就是 FriClaw 当前架构最核心的特征：

**它不是靠某一个超强模块完成全部工作，而是靠多个边界清晰的层次，把一个通用模型组织成一个可以稳定工作的系统。**

如果要用一句话来概括 FriClaw 当前的架构设计，我会说：

**它真正解决的，不是如何让模型回答一句话，而是如何让一个模型在真实系统里长期、稳定、可扩展地工作。**
