# 提案：重构 Cron 定时任务系统

> 创建时间：2026-03-27
> 完成时间：2026-03-28

## 背景

当前的 cron 定时任务实现存在多个架构问题：

1. **低效的轮询方式**：使用 60 秒间隔检查 cron 表达式是否匹配，浪费 CPU 资源
2. **简陋的 cron 解析器**：仅支持基础模式，缺少步长值（`*/5`）、范围和复杂表达式
3. **基于文件的持久化**：使用 JSON 文件存储而非 SQLite，与项目的数据库选择不一致
4. **紧耦合**：调度器直接依赖 Dispatcher，导致测试和复用困难
5. **可观测性不足**：没有日志、指标或执行历史追踪
6. **不支持时区**：所有任务都在服务器时区运行，对多地区用户不友好
7. **MCP 架构问题**：MCP 进程独立运行 scheduler，主进程无法收到事件

## 目标

- [x] 使用成熟的库替换轮询机制，实现高效的 cron 计算
- [x] 将持久化从 JSON 文件迁移到 SQLite，使用合理的数据库模式
- [x] 使用基于事件的架构解耦调度器和消息分发器
- [x] 添加执行历史和审计追踪
- [x] 支持时区感知的任务调度（默认 Asia/Shanghai）
- [x] 提升测试覆盖率和可维护性
- [x] 支持一次性任务（ISO 时间字符串）
- [x] 修复 MCP 架构问题（信号文件通信）

## 非目标

- 兼容旧的 JSON 文件格式（推倒重建，不考虑历史包袱）
- 跨多实例的分布式调度（未来工作）
- 任务管理的 Web UI（独立功能）
- 高级特性如任务依赖或工作流

## 解决方案概述

将 cron 系统重构为更清晰的架构：

1. **使用 `croner` 库**进行健壮的 cron 解析和调度
2. **SQLite 数据库模式**存储任务和执行历史
3. **事件发射器模式**解耦任务执行和消息分发
4. **时区支持**通过任务级别的时区配置（默认 Asia/Shanghai）
5. **执行历史表**用于审计和调试
6. **一次性任务**支持 ISO 时间字符串
7. **信号文件通信**MCP 通过 `.cron-reload` 文件通知主进程重载任务

## 架构改进

### MCP 与主进程分离
- MCP 进程只负责数据库操作（CronStorage）
- 主进程负责所有任务调度（CronScheduler）
- 通过信号文件实现进程间通信

### 消息发送机制
- 定时任务触发时查找对应 Gateway
- 动态创建 reply 函数发送消息到正确平台
- 支持飞书、企业微信、微信、Dashboard
