MODULE 01

封面 / 总览

将原先 Hero 内容合并进首个模块，以封面视角概览 FriClaw 的角色定位、路径主线与整体架构切面。

FriClaw

Friday + Claw。它不是单一问答机器人，而是一套从消息接入、调度编排、会话上下文、Agent 执行，到能力扩展与系统收束的工程化 AI 助手体系。

平台入口

飞书、企业微信、Dashboard 等入口统一落到同一消息模型，外部渠道差异被隔离在接入层。

核心主线

消息进入系统后按层推进：接入 → 调度 → 会话上下文 → Agent 执行 → 能力扩展 → 全局收束。

运行方式

同一会话内强调串行与可追踪，不同会话之间保留并发吞吐，确保稳定与可运维性。

展板目标

本页聚焦架构关系，而非教程导学；重点是帮助工程师快速理解责任边界与系统连接方式。

总览要点

  • 统一入口不同平台以统一消息契约进入内核。
  • 层次清晰每一层只承担本层职责，降低改动扩散。
  • 系统化视角关注结构、流向、边界与演进空间。
MODULE 02

接入层

负责把平台协议转换为内部一致的输入事件，是系统的边缘适配带。

边缘接入拓扑

平台网关

飞书、企业微信、Dashboard 等入口处理鉴权、平台格式、回包能力与连接形态。

统一消息模型

抽象出平台、chatId、userId、内容、附件和元数据，让后续层无需感知渠道差异。

边缘约束

屏蔽平台差异，但保留必要上下文，例如群聊/私聊、消息来源、附件类型与回调句柄。

工程收益

新增一个平台主要落在接入层，不需要重写调度与 Agent 主逻辑。

接入层职责

  • 协议适配把外部消息协议映射成内部事件。
  • 身份透传保留用户、会话、群组等关键标识。
  • 响应封装将执行结果回写为平台可消费格式。
MODULE 03

调度层

调度层是消息总线与执行编排入口，负责分流、排队和生命周期控制。

消息编排路径

消息到达
→
命令识别
→
会话归属
→
队列排程
Dispatcher

统一承接接入层消息，决定走命令路径还是正常对话路径。

Lane Queue

以会话为粒度串行化处理，避免同一上下文出现竞争执行。

流式转发

为后续执行层保留流式回传通道，让前端或平台能持续看到处理进度。

关闭策略

在停止接收新任务与等待已有队列清空之间建立可控的关停边界。

调度层要点

  • 路由决定消息走哪条系统路径。
  • 排队确保同会话处理顺序稳定。
  • 编排把上下游组件串成可运营链路。
MODULE 04

会话与上下文层

这一层决定系统如何记住谁在说话、此前发生了什么、当前执行所需的上下文应如何装配。

上下文组装面板

Session Manager

为 platform + chatId 等标识建立可复用的会话实体，管理工作目录和对话状态。

Identity / Knowledge / Episode

静态身份、长期知识与会话经历共同构成多层上下文来源。

检索注入

按请求需要装配相关内容，而不是将所有记忆粗暴堆入提示词。

上下文边界

这一层控制注入范围、粒度与顺序，直接影响成本、准确性与稳定性。

会话层要点

  • 会话隔离不同聊天保持独立上下文空间。
  • 多层记忆人格、知识、经历承担不同信息职责。
  • 按需组装控制上下文规模与相关度。
MODULE 05

Agent 执行层

真正承担推理与操作的是 Agent 执行层，它负责驱动 Claude Code 子进程、消费事件流并组织最终回复。

执行循环

请求构造

把用户输入、会话上下文、能力定义整理成执行请求，发往 Agent 运行单元。

子进程复用

通过可复用进程或工作区减少重复冷启动，提升多轮对话吞吐。

事件流解析

从 thinking、text、tool_use、done 等事件中还原执行过程与最终结果。

结果回传

将执行产物重新映射到调度链路，供平台侧持续流式展示或一次性回复。

执行层要点

  • 推理核心承担模型调用与步骤推进。
  • 过程可见通过事件流暴露执行状态。
  • 结果可交付输出回复、调用记录与收束信号。
MODULE 06

能力扩展层

系统不仅要回答问题，还要调用工具、触发任务并与外部能力生态连接，这一层负责扩展可做的事。

能力挂载矩阵

MCP 服务

通过标准协议接入搜索、文件、知识、网关发送等可组合能力，形成统一扩展面。

Cron / 主动任务

定时与延迟执行让系统具备主动性，不再局限于收到消息后被动响应。

配置热更新

能力清单可随着会话或运行时变化而重载，降低主进程重启成本。

边界治理

能力扩展越强，越要清楚约束权限、执行成本、故障回退和可观测性。

扩展层要点

  • 工具化把外部系统能力变成可调用接口。
  • 主动化支持定时、提醒与异步任务闭环。
  • 可演进新增能力不必重写主流程。
MODULE 07

全局收束

最终从系统视角收束所有层级关系，回答这套架构为什么成立、如何维护、以及演进时应优先守住什么。

全局架构闭环

主链路闭环

接入层负责收口，调度层负责组织，会话层负责理解，执行层负责产出，扩展层负责放大能力。

核心设计原则

统一模型、会话隔离、串行保障、流式反馈、能力外置，是这一架构最重要的工程原则。

演进方向

后续优化通常落在观测性、缓存、调度策略、工具治理与跨平台一致性，而不是重新拆穿主链路。

维护重点

优先守住职责边界和数据契约，避免让接入、调度、上下文与执行逻辑互相渗透。

收束结论

  • 一条主轴七个模块沿同一工程主轴展开，不再混入教程型杂项内容。
  • 一个页面整页保留导航、进度、滚动与 reveal，但主体严格收敛为七个顶层模块。
  • 一种视角从工程师视角阅读系统，而不是从新手学习视角拆分交互教学。
  • 一个结论FriClaw 的关键优势在于层次分明、可扩展且可维护的执行架构。
