BACKGROUND

背景

这一部分不讲实现细节，先讲清楚 FriClaw 为什么值得做，以及它解决的到底是什么问题。

Why FriClaw

FriClaw

OpenClaw 真正重要的地方，不只是能力强，而是它第一次让更多人清楚看到： 大模型可以被组织成一个真正进入工作流、持续帮人做事的 Agent 系统。

FriClaw 的核心判断

没必要从零再造一个完整 Agent。更合理的方式，是基于 Claude Code 这个已经足够强的执行内核，补上外部接入、上下文管理和按需扩展能力。

站在 Claude Code 之上 不是复刻 OpenClaw 组织成真正可工作的系统

OpenClaw 带来的四个关键启发

01 接入真实沟通场景

从本地命令行走向外部 IM，而不是只停留在单机环境里。

02 支持持续工作

不是一次性问答，而是会话可以延续、任务可以接着做。

03 具备上下文连续性

不只看眼前一句话，而是能带着历史和环境继续执行。

04 能力可以持续扩展

通过 Skill 和 MCP，把能力沉淀成可复用、可组合的系统部件。

这一页真正想说明的是

FriClaw 关注的重点，不是把模型做成“更会聊天”，而是把模型组织成一个能够进入团队工作流、长期稳定运行的系统。

FriClaw 的三个设计决策

Constraint 不直接照搬 OpenClaw

OpenClaw 很强，但权限要求和系统复杂度都更高。对个人工作环境来说，可控性和边界感同样重要。

Decision 复用 Claude Code 作为执行内核

既然核心执行能力已经具备，就把精力放在真正缺的部分：IM 接入、会话承接、上下文管理，以及后续扩展机制。

Positioning 把 FriClaw 定位为 co-worker

它不是“另一个聊天机器人”，而是一个可以进入日常工作流、持续协作、持续干活的系统型 Agent。

网关层

消息从哪里来、以什么格式进入系统。

调度层

消息应该如何排队、如何分发、按什么顺序执行。

会话层

消息归属到哪个会话、上下文如何承接、状态如何隔离。

Agent 执行层

谁来理解任务、调用能力、完成执行。

Memory

系统如何把历史、记忆和上下文组织起来，支撑 Agent 持续工作。

Skill / MCP

Agent 在执行过程中可以调用哪些原生能力和扩展能力。

SECTION 02

二、网关层：先统一入口，再谈后面的智能处理

网关层之所以放在最前面，是因为一个真正要进入工作流的 Agent 系统，第一步不是“模型怎么回答”，而是“外部消息怎么稳定进入系统”。

网关层核心职责只有四件事

建立与外部平台的连接
接收平台事件
把平台原始消息转换成统一结构
把系统内部响应翻译回平台原生格式

统一抽象、保留差异

飞书

可以用卡片更新实现更自然的流式输出；飞书话题群里的 thread 可以映射成更细粒度的会话。

企业微信

没有同样的流式卡片能力，就采用分块消息来模拟流式效果。

不同平台

对交互式问题、图片、富文本的支持程度也不同。

文档中的边界与价值

  • 边界

    平台差异留在网关层，系统核心只消费标准消息。

  • 价值 1

    新增入口时，不需要重写 Agent 核心逻辑。

  • 价值 2

    调整平台交互方式时，不需要改动会话和记忆系统。

  • 价值 3

    核心执行层只处理统一消息，不处理平台特有细节。

SECTION 03

三、调度层：让消息处理变得有秩序

调度层主要由两个部分组成：Dispatcher 负责把外部消息纳入系统内部处理流程；Lane Queue 负责保证同一会话内的消息按秩序执行。

Dispatcher：系统里的调度中枢

根据消息归属定位处理目标
把消息投递到对应会话的处理通道
对部分系统命令做前置短路处理
保证外部平台不会直接触碰内部实现

Lane Queue：每个会话有自己的处理泳道

等待
等待
同一会话内：严格串行 不同会话间：完全并行

如果没有调度机制，很容易出现

上下文串线
回复顺序错乱
后发消息覆盖前一轮状态
同一个会话并发触发多个 Agent 执行过程
调度层的价值

把会话处理过程从概率正确，变成机制正确。

关键判断

模型再聪明，也不应该承担消息乱序带来的系统性问题。

SECTION 04

四、会话层：把消息组织成可持续的工作上下文

会话层主要由 SessionManager 承担。它的意义在于：系统处理的不是一条条彼此独立的 message，而是一段持续存在、带有状态、可恢复的 session。

真正有意义的处理单位不是 message，而是 session

会话层真正解决的不是“存一下聊天记录”

状态连续性

FriClaw 才能把一次次消息处理，组织成一个持续工作的过程。

稳定承载点

给后面的 Agent 执行、记忆供给和工作空间隔离，提供一个稳定的承载点。

SECTION 05

五、Agent 执行层：Claude Code 在这里不是聊天接口，而是执行引擎

在当前架构里，Claude Code 承担的角色，并不是一个单纯的回答器，而是会话级执行引擎。

Agent 层希望具备的持续工作能力

在同一会话里延续上下文
在多轮交互里保持状态连续
在需要时调用工具
以流式方式输出内容
在一轮结束后保留可恢复的执行轨迹

Agent 输出的事件流

thinking

不是一次性文本。

text

持续发出执行事件。

tool_use

Agent 当成引擎。

done

执行过程本身可见。

一个 conversationId 会尽量复用同一个 Claude Code 执行上下文

conversationId

尽量复用同一个 Claude Code 执行上下文。

session_id

维持恢复能力。

重点

为了让同一个会话在多轮交互中保持连续性。

工作续接摘要

场景 A：上下文过大，主动压缩

当估算会话上下文接近 Claude Code 最大 token 的 0.7 时，先生成工作续接摘要，再结束旧的执行上下文，并在新的 Claude Code 进程里基于摘要继续。

场景 B：session 过期，被动恢复

当 session 被回收、Claude Code 子进程因 idle timeout 释放，甚至 daemon 重启导致原执行上下文消失时，系统读取最近一次工作续接摘要，重新拉起新的 Claude Code 进程，并把摘要注入进去继续服务。

更准确地说

在原上下文可复用时保持连续性，在原上下文不可复用时通过摘要续接把工作平滑转移到新的执行上下文里。

从架构上看

不是“接入一个 Claude”

FriClaw 做的并不是“接入一个 Claude”。

而是

把 Claude Code 变成系统中的一个可持续运行、可恢复、可扩展的智能执行单元。

SECTION 06

六、记忆与上下文层：让系统不是只看这句话，而是带着状态继续工作

FriClaw 不是把上下文简单理解成聊天记录，而是把记忆拆成多层，并把“上下文”提升成一套可管理的系统能力。

FriClaw 把记忆拆成了三层

Identity（SOUL.md）

系统身份与长期行为基调。

Knowledge

用户知识、偏好、项目相关信息。

Episode

历史会话摘要与经验沉淀。

Identity：定义长期稳定的助手行为基线 Knowledge：沉淀用户和任务相关的长期事实 Episode：压缩和回收历史会话经验

这套记忆系统由三部分共同组成

SQLite

作为记忆索引和全文检索后端。

Markdown 文件

作为记忆的原始存储形式。

MCP tools

作为 Claude Code 访问记忆的接口层。

不是所有信息都应该以同样方式保存
比起“记得更多”，更重要的是“记得更有结构”

上下文层和会话层是相互配合的

会话层

负责定义“这一轮属于哪个连续过程”。

记忆层

负责决定“这个连续过程在更长时间尺度上可以继承什么”。

当前轮次

不脱离眼前问题。

同一会话

不丢状态。

跨会话

又能沉淀长期记忆。

结果

这比单纯把聊天历史交给模型，要稳定得多，也更像一个真正长期工作的系统。

SECTION 07

七、总结：一条请求如何穿过整套架构

主链路的 9 个步骤，以及最后那句最核心的架构概括。

文档中的 9 步主链路

1. 用户在 企业微信 发来消息
↓
2. 对应网关接收消息，并转换成系统统一消息结构
↓
3. Dispatcher 接管这条消息，定位或创建对应会话
↓
4. 消息进入该会话的 Lane Queue，保证会话内串行处理
↓
5. SessionManager 为这次处理提供会话环境与工作空间
↓
6. Claude Code Agent 在该会话上下文中执行本轮任务
↓
7. 执行过程中按需读取记忆、使用工具、调用 MCP 能力
↓
8. Agent 产出流式事件或最终结果
↓
9. 网关把结果翻译回平台原生形式并返回给用户

每一层只做自己该做的事情

网关负责适配平台
→
调度层负责建立秩序
→
会话层负责提供连续性
→
Agent 层负责理解与执行
→
记忆层负责补充长期上下文
扩展层负责提供外部能力
→
多个边界清晰的层次
→
把一个通用模型组织成
→
一个可以稳定工作的系统
→
长期、稳定、可扩展地工作

一句话概括

它真正解决的，不是如何让模型回答一句话，而是如何让一个模型在真实系统里长期、稳定、可扩展地工作。
