# FriClaw 技术架构分享

## 背景

FriClaw 这个项目，是从 OpenClaw 带来的启发开始的。

如果从最表层去看 OpenClaw，它最打动人的地方，并不是某一个单点功能，而是它已经把一个 Agent 系统最关键的几件事串起来了：

- **能够接到外部 IM**：不是只能在本地命令行里工作，而是能进入真实沟通场景
- **能够持续工作**：不是一次性问答，而是会话可以延续、状态可以承接
- **有更好的上下文能力**：不是只看眼前一句话，而是能带着历史和环境继续执行
- **支持 Skill 扩展**：能力不是写死的，而是可以不断增加、沉淀和组织

也正因为如此，OpenClaw 给我们的启发并不是“它实现了哪些功能”，而是它已经展示出一种很有价值的方向：**如何把一个通用大模型，组织成一个能真正跑在工作流里的 Agent 系统。**

但对我们来说，直接使用 OpenClaw 并不是最终目标。一个很现实的原因是，OpenClaw 整体能力非常强，但也因此显得偏重：它更像一套完整的 Agent 运行环境，而不是一个可以轻量接入的本地工具。与此同时，它需要较深地进入工作环境，而在真实办公场景里，安全边界始终是一个绕不过去的问题——尤其当它需要安装在自己的工作机上时，我们很难在缺乏足够安全把握的前提下直接把它真正用起来。

也正因为如此，我们真正想做的，并不是简单“把 OpenClaw 装起来”，而是**构建一个我们自己需要的 OpenClaw**。

这个“我们需要的 OpenClaw”，重点不在于简单复刻，而在于围绕自己的使用场景、接入方式和扩展需求，把那套核心思路重新组织成一套更适合当前项目的架构。我们需要的，不只是一个能聊天的 Agent，而是一个：

- 可以接入飞书、企业微信、微信 等入口的系统
- 可以按会话稳定运行、持续承接上下文的系统
- 可以把长期记忆、工作空间和当前任务状态组织起来的系统
- 可以通过 MCP、Skill 和配置机制不断扩展能力的系统

所以，FriClaw 当前架构的出发点，其实不是“从零设计一个全新的 AI 框架”，而是沿着 OpenClaw 证明过的那条方向继续往前走：把外部接入、持续执行、上下文管理和能力扩展这几件事，做成一套真正适合我们自己的系统。

也正因为如此，FriClaw 当前的架构，本质上不是围绕“单次问答”组织的，而是围绕“一个Agent系统如何稳定运行”来组织的。

从当前项目的设计上看，它大致被拆成了这样几层：

```text
外部平台
  ↓
网关层（Feishu / WeCom / WeChat）
  ↓
调度与会话层（Dispatcher / SessionManager / Lane Queue）
  ↓
Agent 执行层（Claude Code Agent）
  ↓
记忆与上下文层（Identity / Knowledge / Episode / Session Context）
  ↓
扩展能力层（MCP / 配置注入 / 热重载）
```

这几层并不是简单地把代码分目录，而是在回答不同层次的问题：

- 网关层解决“消息从哪里来、以什么形式进来”
- 调度与会话层解决“消息应该按什么粒度被组织和处理”
- Agent 层解决“谁来理解任务并执行它”
- 记忆层解决“系统如何不只依赖当下这句话，而是带着上下文继续工作”
- 扩展层解决“这套系统以后怎么继续长”

下面我按这个分层，展开说说 FriClaw 当前架构的设计思路。

---

## 一、整体架构总览：先把入口、执行和记忆拆开

如果从宏观上看，FriClaw 的结构其实很清楚：它不是一个单体聊天机器人，而是一条从“消息接入”到“会话组织”再到“Agent 执行”的链路。

```text
飞书 / 企业微信 / Dashboard
        ↓
     Gateway
        ↓
    Dispatcher
        ↓
 SessionManager + Lane Queue
        ↓
 Claude Code Agent
        ↓
 Memory / MCP / Workspace
        ↓
   响应回到原平台
```

这里最重要的架构判断有两个。

### 1. 先把“接入”与“智能”拆开

FriClaw 不是直接让飞书、企业微信去碰 Claude Code，也不是让 Agent 自己处理平台细节。平台接入被放在最外层，由各自网关负责；智能执行被放在中间，由 Agent 负责。这意味着平台差异不会一路渗透到核心执行层。

这个拆分的价值很现实：

- 新增一个平台接入时，不需要重写 Agent 核心逻辑
- 更换响应样式时，不需要改动会话和记忆系统
- 核心层只处理统一消息，不处理平台特有协议细节

### 2. 先把“运行系统”搭起来，再让模型进去工作

很多 AI 项目一开始最容易做的，是先把模型接起来；但 FriClaw 当前架构更强调的是：**先把消息如何排队、会话如何隔离、状态如何沉淀这些运行时问题定义清楚，再把模型放进来。**

也正因为如此，这套系统虽然以 Agent 为核心，但并不是“以模型为中心”的结构，而更像“以运行机制为骨架，模型在其中承担推理和执行角色”的结构。

---

## 二、网关层：先统一入口，再谈后面的智能处理

FriClaw 最外层是网关层。它的职责不复杂，但非常关键：**把不同平台的消息接入方式、事件格式和回复机制，收敛成系统内部可以统一处理的标准消息。**

从当前设计看，项目主要有三类入口：

- 飞书网关
- 企业微信网关
- Dashboard 入口

它们的共同点是都位于系统边界，负责和外部通信；它们的差异则来自平台能力本身。

### 1. 网关层不做业务判断，只做平台适配

这是 FriClaw 当前架构里一个很好的边界控制。

网关层并不负责理解用户意图，也不负责管理复杂上下文。它主要做这几件事：

- 建立与外部平台的连接
- 接收平台事件
- 把平台原始消息转换成统一结构
- 把系统内部产生的响应再翻译回平台原生格式

这个边界看起来很“朴素”，但其实很重要。因为一旦让平台层承担太多业务判断，系统很快就会出现两个问题：

- 平台差异开始污染核心逻辑
- 后续新增入口时，业务逻辑会被复制一遍

FriClaw 把这件事控制得比较清楚：**平台差异留在网关层，系统核心只消费标准消息。**

### 2. 统一入口，不等于抹平差异

虽然网关层的目标是统一入口，但它并不是粗暴地把所有平台都“做成一样”。相反，当前设计里其实保留了对平台特性的尊重。

比如：

- 飞书可以用卡片更新做更自然的流式输出
- 企业微信不具备同样的流式卡片能力，就采用分块消息模拟流式效果
- 飞书话题群里的 thread 可以映射为更细粒度的会话
- 不同平台对交互式问题、图片、富文本的支持程度也不同

也就是说，网关层做的不是“消灭平台差异”，而是“吸收平台差异”。

这件事对整体架构很关键。因为一个真正面向多平台的助手系统，不能假设所有平台都是同构的。FriClaw 当前设计更合理的地方在于：它允许外层保留差异，但不让这些差异继续往内层扩散。

### 3. 网关层的意义，不只是接消息

从更高一层看，网关层其实在做两件事：

- 对外，它决定 FriClaw 能出现在哪里
- 对内，它决定后续系统能否建立统一处理链路

所以这一层虽然不“智能”，但它决定了整个系统是不是可接入、可扩展、可持续演进的。

---

## 三、调度与会话层：让消息变成“可控的会话处理过程”

如果说网关层解决的是“消息进来”，那么调度与会话层解决的就是另一个更核心的问题：**消息进来之后，系统应该按什么单位来处理它。**

FriClaw 当前设计里，这一层主要由三个概念构成：

- Dispatcher
- SessionManager
- Lane Queue

这三者组合起来，实际上定义了整个系统的运行节奏。

### 1. Dispatcher：系统里真正的中枢，而不是消息转发器

从表面上看，Dispatcher 像是在做“转发”：收到消息，找到会话，再交给 Agent。但如果只把它理解成转发器，会低估它在架构里的位置。

Dispatcher 真正的价值在于，它把“外部消息事件”转换成“内部任务处理流程”。

换句话说，在网关层拿到的还是平台消息；到了 Dispatcher 这一层，消息开始被纳入 FriClaw 自己的运行秩序里。

它负责的事情包括：

- 根据消息归属定位或创建会话
- 把消息投递到对应会话的处理通道
- 对某些系统命令做前置短路处理
- 保证外部平台不会直接触碰会话内部实现

这个设计的好处是，系统从这一层开始，才真正形成了自己的内部协议。

### 2. SessionManager：把连续对话从“消息列表”提升成“会话对象”

一个助手系统和普通消息回调程序最大的区别，就是它不能把每条消息都当成独立请求。

FriClaw 当前设计里，会话不是附属概念，而是主组织单元。SessionManager 的存在，意味着系统处理的不是零散消息，而是一段持续存在、带有状态、可恢复的会话。

这背后其实有一个很重要的架构判断：

**对助手系统来说，真正有意义的处理单位不是 message，而是 session。**

因为只有引入会话这个层次，后面这些事情才有落点：

- 上下文如何延续
- 哪个工作空间归属于哪个用户/聊天
- 哪个 Claude 子进程应当继续复用
- 哪些历史状态应当跟着本轮继续生效

如果没有会话层，整个系统很快就会退化成“每条消息调用一次模型”的松散结构，这显然支撑不了 FriClaw 这种目标。

### 3. Lane Queue：真正保证系统稳定运行的，不是模型，而是调度秩序

FriClaw 当前架构里，一个很值得拿出来讲的点就是 Lane Queue。

它解决的问题不是“怎么调用模型”，而是“怎么避免同一会话里的消息互相打架”。

这个问题在聊天型系统里非常真实：

- 用户连续发送多条消息
- 平台重复投递或短时间内多次触发
- 一条消息还没处理完，下一条又到了

如果没有调度机制，很容易出现：

- 上下文串线
- 回复顺序错乱
- 后发消息覆盖前一轮状态
- 同一个会话并发触发多个 Agent 执行过程

Lane Queue 的设计，本质上是在系统层引入一个简单但非常有效的约束：

```text
同一会话内：严格串行
不同会话间：完全并行
```

这个约束决定了 FriClaw 的运行方式不是“谁先抢到资源谁执行”，而是“每个会话有自己的处理泳道”。

从架构角度看，这个设计非常重要，因为它把并发问题局部化了：

- 系统整体仍然可以并行处理很多用户
- 但单个用户/单个会话内部，又是稳定、可预期的

这比用一个全局锁更合理，也比完全放开并发更适合会话型 Agent 系统。

### 4. 调度层的本质：不是为了排队，而是为了建立确定性

很多时候我们会把队列机制理解成“削峰填谷”或者“控制并发”，但对 FriClaw 来说，Lane Queue 更大的价值其实是：**把会话处理过程从概率正确，变成机制正确。**

也就是说，模型再聪明，也不应该承担消息乱序带来的系统性问题；这种问题应该先由架构解决。

从这个角度看，调度与会话层其实是在替 Agent 创造一个更干净的运行环境。只有这一层把秩序建立好了，后面的智能执行才不会建立在不稳定的输入之上。

---

## 四、Agent 执行层：Claude Code 在这里不是聊天接口，而是执行引擎

到了 Agent 层，FriClaw 才真正进入“智能处理”的部分。

但这里有一个很重要的区别：在当前架构里，Claude Code 承担的角色，并不是一个单纯的回答器，而是**会话级执行引擎**。

这也是 FriClaw 和很多简单 AI 接入方案最大的不同之一。

### 1. Agent 层解决的，不只是回答问题

如果系统目标只是“一问一答”，那么直接请求模型、拿到文本、返回平台，其实就够了。

但 FriClaw 当前设计显然不是这样。它希望 Agent 具备的是更持续的工作能力：

- 在同一会话里延续上下文
- 在多轮交互里保持状态连续
- 在需要时调用工具
- 以流式方式输出内容
- 在一轮结束后保留可恢复的执行轨迹

所以 Claude Code Agent 在这里更像一个**长期驻留的会话执行单元**，而不是“调用一次就结束”的 API 包装层。

### 2. 一个会话对应一个执行上下文

当前设计中，一个 conversationId 对应一个 Claude Code 子进程，并通过 session_id 维持恢复能力。这种做法的核心意义不是“实现复杂”，而是为了保持会话连续性。

因为对 FriClaw 来说，真正重要的不是每轮都拿到一个答案，而是让同一个会话在多轮交互中仍然像“同一个助手”在持续工作。

从这个角度看，Agent 层做的不是“临时生成响应”，而是“持续承载会话执行”。

这带来几个很直接的架构收益：

- 同一会话的上下文不会反复丢失
- 多轮交互的状态切换更自然
- 工具调用与追问可以被组织成连续过程
- 系统可以更稳定地管理长会话，而不是每轮都重新拼装一切

### 3. 流式交互是执行层的一部分，不只是展示优化

FriClaw 当前设计里，Agent 输出不是一次性文本，而是会产生 thinking、text、tool_use、done 等事件流。

这件事很值得强调，因为它说明系统并不是把 Agent 当成“黑箱返回最终字符串”，而是把它当成一个持续发出执行事件的引擎。

一旦用这种方式组织，前后的架构关系就会变得更清楚：

- 网关层负责把这些事件翻译成平台可展示的形式
- 调度层负责保证事件属于正确的会话
- Agent 层负责产生执行过程本身

也就是说，流式并不是 UI 层的小优化，而是整个系统执行模型的一部分。

### 4. Claude Code 在 FriClaw 里，更像一个“可被系统组织的智能核心”

FriClaw 当前架构没有让 Claude Code 单独暴露在系统中心，而是把它放进了一个更完整的运行框架里：

- 前面有会话和调度机制约束输入
- 后面有记忆、工作空间和 MCP 能力提供支撑
- 外面还有多平台网关负责承接与呈现

所以从架构上看，FriClaw 做的并不是“接入一个 Claude”，而是**把 Claude Code 变成系统中的一个可持续运行、可恢复、可扩展的智能执行单元。**

---

## 五、记忆与上下文层：让系统不是只看这句话，而是带着状态继续工作

如果说 Agent 层解决的是“谁来执行”，那么记忆与上下文层解决的就是“执行时凭什么知道自己现在处在什么状态”。

这一层是 FriClaw 当前架构里非常关键的一层，因为助手系统和普通聊天系统最大的差异，往往就体现在这里。

### 1. FriClaw 不是把上下文简单理解成聊天记录

在很多简单的聊天产品里，“上下文”基本等于历史消息拼接。但在 FriClaw 当前设计中，这种理解是不够的。

因为一个真正可工作的助手系统，不只是需要记住用户刚刚说过什么，还需要记住一些更长期、更稳定、对未来仍然有价值的东西。

从当前设计上看，FriClaw 把记忆拆成了三层：

- **Identity**：系统身份与长期行为基调
- **Knowledge**：用户知识、偏好、项目相关信息
- **Episode**：历史会话摘要与经验沉淀

这种分层方式背后的思路很清楚：

- 不是所有信息都应该以同样方式保存
- 不是所有历史都值得每次完整重放
- 比起“记得更多”，更重要的是“记得更有结构”

### 2. Identity 层：先确定这个系统是谁

Identity 层听起来很抽象，但它在助手系统里其实非常重要。

因为 FriClaw 不是一个裸模型，而是一个带有明确身份设定的系统。它的风格、角色、行为边界，并不是每轮由模型即兴决定，而是由更稳定的身份信息持续约束。

这一层的意义不在于“增加人设感”，而在于给整个系统建立长期稳定的行为基线。

换句话说，它解决的是：

- 这个助手应该以什么方式工作
- 它面对用户时的默认行为倾向是什么
- 哪些原则应该跨会话持续生效

### 3. Knowledge 层：沉淀的是用户相关事实，而不是一次性聊天噪声

Knowledge 层更偏向长期有效信息，比如用户偏好、联系人、项目背景、一些会被反复用到的知识。

这一层很关键，因为它把“会话记忆”往前推进了一步：不再只是记住这次说过什么，而是开始沉淀“以后可能继续有用的信息”。

从系统角度看，这意味着 FriClaw 不只是一个会聊天的 Agent，而是一个可以逐渐积累用户知识的系统。

这也是它能从一次次对话里慢慢形成稳定工作方式的基础。

### 4. Episode 层：保留历史，但不让历史无限膨胀

长期会话系统一定会遇到一个问题：历史会越来越长。

如果所有历史都原样堆给模型，会遇到几个非常现实的问题：

- 成本持续上升
- 噪声越来越多
- 当前真正重要的信息反而被淹没

FriClaw 当前设计没有走“完整重放所有历史”的路线，而是通过 Episode 这种摘要化的方式保存会话阶段性结果。

这个思路很合理。因为对长期运行的助手来说，最有价值的往往不是原始聊天记录本身，而是从聊天中提炼出来的任务结果、关键信息和阶段结论。

### 5. 记忆层真正解决的，是“可持续的上下文供给”

把三层记忆放在一起看，会更容易理解这层的真实作用。

```text
Identity：定义长期稳定的助手行为基线
Knowledge：沉淀用户和任务相关的长期事实
Episode：压缩和回收历史会话经验
```

这意味着 FriClaw 在给 Agent 提供上下文时，不是只依赖短期聊天历史，而是可以把长期身份、已沉淀知识和历史摘要一起组织起来。

这件事的架构价值很大，因为它把“上下文”从一段临时文本，提升成了一套可管理的系统能力。

### 6. 上下文层和会话层是相互配合的

值得注意的是，记忆层并不是孤立工作的。

- 会话层负责定义“这一轮属于哪个连续过程”
- 记忆层负责决定“这个连续过程在更长时间尺度上可以继承什么”

两者配合之后，FriClaw 才真正具备一种助手系统该有的连续性：

- 当前轮次不脱离眼前问题
- 同一会话不丢状态
- 跨会话又能沉淀长期记忆

这比单纯把聊天历史交给模型，要稳定得多，也更像一个真正长期工作的系统。

---

## 六、扩展能力层：让系统不是写死的，而是可以继续长的

如果说前面几层解决的是“先把系统跑起来”，那么扩展能力层解决的就是另一个长期问题：**FriClaw 以后怎么继续长。**

当前设计里，这部分主要围绕 MCP、工作空间配置注入和热重载机制展开。

### 1. MCP 的意义，不只是接工具

在很多项目里，工具调用容易被做成“在主工程里堆一组函数，然后让模型去调”。这种做法短期当然有效，但随着能力变多，很容易出现边界混乱、配置散落、扩展困难的问题。

FriClaw 当前架构选择用 MCP 来承接工具扩展，本质上是在做一件更长期的事情：**把能力扩展从代码耦合，转成协议化接入。**

这意味着，工具能力不再只是主工程里的内部细节，而是开始具备更清晰的接口边界。

从当前设计看，MCP 主要承担几类能力：

- 记忆系统能力
- 定时任务能力
- 工作空间相关能力
- 未来更多外部服务接入能力

这套设计的好处是，FriClaw 不需要把所有能力都硬编码到 Agent 本体里，而是可以让 Agent 在一个相对稳定的执行核心之外，持续挂接新的工具能力。

### 2. 工作空间注入：让每个会话拥有自己的运行环境

FriClaw 当前设计里，工作空间不是附属物，而是整个系统运行的重要基础。

每个会话会拥有自己的工作空间目录，其中承载：

- 会话元数据
- 上下文信息
- 可用工具配置
- MCP 配置
- 记忆与缓存相关内容

这个设计背后的逻辑很清楚：

**Agent 的运行不应该只是“拿一段输入去推理”，而应该发生在一个有明确边界的会话环境里。**

有了工作空间之后，很多事情就都有了承载点：

- 哪些配置属于这个会话
- 哪些工具在这个上下文里可用
- 哪些临时状态应该跟着这个会话走
- 哪些内容在会话结束时可以清理或保留

这让 FriClaw 的 Agent 执行更像“在一个隔离环境里持续工作”，而不是在全局共享状态里四处借数据。

### 3. 热重载的价值，在于让扩展成本下降

扩展机制真正好不好，不只看能不能扩，而要看扩的时候是否会把系统打断。

FriClaw 当前设计里，一个很好的点是：MCP 配置和扩展能力的生效，尽量通过新会话创建时的配置注入来完成，而不是要求整个主进程频繁重启。

这种热重载思路带来的好处很实际：

- 新能力上线成本更低
- 已在运行中的会话不容易被打断
- 配置修改对系统的影响范围更容易控制

换句话说，这一层真正解决的是：FriClaw 不是一个功能定死的成品，而是一套可以持续增量演进的运行框架。

---

## 七、一条请求如何穿过整套架构

如果把前面几层重新串起来，FriClaw 当前架构的主链路其实可以概括成下面这样：

```text
1. 用户在飞书 / 企业微信 / Dashboard 发来消息
2. 对应网关接收消息，并转换成系统统一消息结构
3. Dispatcher 接管这条消息，定位或创建对应会话
4. 消息进入该会话的 Lane Queue，保证会话内串行处理
5. SessionManager 为这次处理提供会话环境与工作空间
6. Claude Code Agent 在该会话上下文中执行本轮任务
7. 执行过程中按需读取记忆、使用工具、调用 MCP 能力
8. Agent 产出流式事件或最终结果
9. 网关把结果翻译回平台原生形式并返回给用户
```

如果只看步骤，这条链路并不复杂；但它真正重要的地方在于，每一层只做自己该做的事情：

- 网关负责适配平台
- 调度层负责建立秩序
- 会话层负责提供连续性
- Agent 层负责理解与执行
- 记忆层负责补充长期上下文
- 扩展层负责提供外部能力

这就是 FriClaw 当前架构最核心的特征：

**它不是靠某一个超强模块完成全部工作，而是靠多个边界清晰的层次，把一个通用模型组织成一个可以稳定工作的系统。**

---

## 八、总结：FriClaw 的重点不是“更会聊天”，而是“更像一个系统”

如果要用一句话来概括 FriClaw 当前的架构设计，我会说：

**它真正解决的，不是如何让模型回答一句话，而是如何让一个模型在真实系统里长期、稳定、可扩展地工作。**

所以这套架构的重点，始终不只是模型能力，而是围绕模型建立起了一整套运行机制：

- 用网关层承接多平台接入
- 用 Dispatcher、SessionManager 和 Lane Queue 建立会话秩序
- 用 Claude Code Agent 承担持续执行
- 用分层记忆提供长期上下文
- 用 MCP 和工作空间机制保证扩展能力

从结果上看，FriClaw 当前项目已经不是一个“给聊天界面接上大模型”的简单系统了。它更像一个围绕 Agent 组织起来的运行框架：模型当然很重要，但真正让它能持续工作的，是外面这整套架构。

而这，可能也是 FriClaw 当前设计最值得分享的地方。
